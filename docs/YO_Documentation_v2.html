YouTube Organizer

Documentação técnica e funcional do YouTube Organizer v2.0.

Introdução

O YouTube é uma das maiores bibliotecas de conhecimento e entretenimento do mundo, mas não oferece nenhuma forma nativa de organizar vídeos salvos em categorias livres, controlar quais já foram assistidos ou fazer backup dessas referências de forma portátil.

O YouTube Organizer resolve isso: uma aplicação leve e privada onde o usuário cria suas próprias categorias, adiciona vídeos por URL e acompanha o que já assistiu, o que quer assistir e o que considera favorito — tudo sem criar conta, sem servidor e sem depender de nenhum serviço externo.

🎯 Objetivo
Resolver o problema de gerenciar grandes volumes de vídeos do YouTube sem uma plataforma dedicada — permitindo categorizar, rastrear status de visualização, favoritar e fazer backup de referências de forma estruturada, rápida e 100% offline.
👤 Público-alvo
Criadores de conteúdo, estudantes, pesquisadores e usuários assíduos do YouTube que precisam organizar e rastrear referências de vídeos de forma local e privada, sem depender de contas ou serviços externos.
ℹ️
Versão do documento Esta é a versão 2.0 da documentação do YouTube Organizer. Última atualização: 05/03/2026 — adicionado RF41, RF42, RF43 (Zona de Perigo) e componente DangerZone.

Overview

A aplicação é uma SPA construída com React 18 + Vite 5, sem backend e sem React Router. O usuário cola uma URL do YouTube, o sistema extrai o ID via regex em utils/videoParser.js e, se uma API Key estiver configurada, busca os metadados (título, canal, thumbnail, duração, views) via YouTube Data API v3. Todos os dados persistem em localStorage e podem ser exportados/importados em JSON a qualquer momento.

A navegação entre as cinco telas — Dashboard, CategoryPage, FavoritesView, WatchedView e UnwatchedView — é controlada por um roteador manual via currentView / currentCategoryId no AppContext, sem dependência de biblioteca de roteamento.

Escopo

O que está dentro e fora do escopo da aplicação:

✅ Dentro do Escopo
  • Criar e gerenciar categorias com nome e cor personalizados
  • Adicionar vídeos por URL (youtube.com e youtu.be)
  • Busca automática de metadados via YouTube Data API v3 (título, canal, thumbnail, duração, views)
  • Marcar vídeos como assistido / favorito
  • Mover e copiar vídeos entre categorias (individual ou em massa)
  • Buscar vídeos por título/canal e ordenar por data, views ou duração
  • Verificar disponibilidade de vídeos (detecta vídeos removidos/privados)
  • Export/import completo dos dados em JSON
  • Auto-backup periódico configurável
  • Tema claro/escuro persistido entre sessões
❌ Fora do Escopo
  • Backend/servidor — sem autenticação, sem banco de dados remoto
  • Sincronização entre dispositivos (sem nuvem)
  • Reprodução de vídeos embutida — apenas links diretos ao YouTube
  • Gerenciamento de playlists reais do YouTube (apenas organização local)
  • Extensão de navegador
  • Aplicativo mobile nativo
  • Suporte a outras plataformas de vídeo (Vimeo, Twitch, etc.)

Quick Start

Como executar o projeto localmente do zero.

Pré-requisitos

  • Node.js ≥ 18.x
  • npm ≥ 9.x ou compatível (pnpm/yarn também funcionam)
  • Navegador moderno com suporte a ES2022+ (Chrome 90+, Firefox 88+, Edge 90+, Safari 14+)

Instalação

1
Clone o repositório Repositório oficial: github.com/pedrolabre/youtube-organizer
Terminal
git clone https://github.com/pedrolabre/youtube-organizer.git cd youtube-organizer
2
Instale as dependências e inicie o servidor
Terminal
npm install npm run dev
3
Abra no navegador O Vite abre automaticamente (open: true). Se não abrir, acesse http://localhost:3000.
4
Configure a API Key (opcional) Para buscar metadados de vídeos automaticamente, configure uma YouTube Data API v3 key nas Configurações da aplicação.
⚠️
Nota sobre API Key A aplicação funciona sem API Key. A chave só é necessária para buscar automaticamente título, duração, thumbnail e outros metadados de vídeos via URL.

Funcionalidades

Lista completa das funcionalidades disponíveis na aplicação.

📁
Categorias
Crie, edite e exclua categorias com nome e cor personalizados. Inclui uma categoria padrão Sem Categoria que não pode ser excluída.
📹
Gerenciar Vídeos
Adicione vídeos por URL (youtube.com ou youtu.be), mova ou copie entre categorias.
🔑
YouTube Data API
Com uma API Key configurada, busca automaticamente título, canal, thumbnail, duração, views, tags e categoria do YouTube.
⭐
Favoritos
Marque vídeos como favoritos (favorite: true). Uma aba dedicada no dashboard lista todos os favoritos independente de categoria.
👁️
Controle de Assistidos
Marque vídeos como assistido/não assistido. O dashboard oferece abas Para Assistir e Histórico com contadores.
⚡
Ações em Massa
Selecione múltiplos vídeos com checkboxes para mover, excluir ou marcar como assistido de uma vez via barra de ações integrada em CategoryActions.
🔍
Busca e Ordenação
Busca por título ou canal dentro de cada categoria. Ordenação por data de adição, data de publicação, views ou duração (crescente/decrescente).
🪄
Magic Categorization
Analisa tags e categoria do YouTube de cada vídeo para sugerir agrupamentos automáticos. Permite buscar metadados faltantes antes da análise.
🔎
Verificação de Disponibilidade
Checa em lote se vídeos ainda estão disponíveis no YouTube, identificando os removidos ou tornados privados (checkVideosAvailability).
💾
Export / Import JSON
Exporte todos os dados ou apenas uma categoria em JSON com nome e data no arquivo. Import com opção de mesclar ou substituir os dados existentes.
🖥️
Auto-Backup
Salva automaticamente em um arquivo local via File System Access API (Chrome/Edge). Mostra horário do último save e status em tempo real.
🌙
Tema Claro / Escuro
Alterna entre tema claro e escuro com transição suave. Preferência persistida no localStorage entre sessões.
⚠️
Zona de Perigo
Três ações destrutivas com recuperação de 10 minutos: apagar todos os vídeos, apagar todas as categorias, ou apagar apenas as categorias vazias. Confirmação inline obrigatória; barra de progresso regressiva; estado sobrevive ao fechar e reabrir o modal.

Requisitos Funcionais

O que o sistema deve fazer — todos os comportamentos e funções obrigatórios derivados diretamente do código.

Categorias

📁RF01AltaCategoriasv1.0
Criar categoria
O sistema deve permitir criar uma categoria informando nome (entre 2 e 50 caracteres, único entre as existentes) e uma cor em hex. O ID é gerado automaticamente no formato cat_{timestamp}_{random}.
✏️RF02AltaCategoriasv1.0
Editar categoria
O sistema deve permitir editar o nome e a cor de uma categoria existente, respeitando as mesmas regras de validação da criação (excluindo a própria categoria da verificação de duplicata).
🗑️RF03AltaCategoriasv1.0
Excluir categoria com escolha de destino dos vídeos
Ao excluir uma categoria, o sistema deve oferecer duas opções: (a) mover os vídeos exclusivos para Sem Categoria; ou (b) excluir permanentemente os vídeos que estão apenas nessa categoria. Vídeos que pertencem a outras categorias simultaneamente apenas perdem o vínculo.
🔒RF04AltaCategoriasv1.0
Proteção da categoria padrão
A categoria Sem Categoria (UNCATEGORIZED_ID) não pode ser excluída nem renomeada. Acumula automaticamente vídeos sem categoria definida.
📊RF05MédiaCategoriasv1.0
Contagem de vídeos por categoria
O dashboard deve exibir a contagem de vídeos de cada categoria, com diferença visual entre assistidos e não assistidos, calculada dinamicamente via getVideoCounts().
↕️RF06MédiaCategoriasv1.0
Ordenar categorias no dashboard
O sistema deve permitir ordenar as categorias exibidas no dashboard por: nome (A–Z / Z–A), data de criação e quantidade de vídeos.

Gestão de Vídeos

🔗RF07AltaVídeosv1.0
Aceitar múltiplos formatos de URL do YouTube
O sistema deve extrair o ID do vídeo a partir dos formatos: youtube.com/watch?v=, youtu.be/, youtube.com/embed/, m.youtube.com/watch?v= e o próprio ID de 11 caracteres diretamente.
⚡RF08AltaVídeosv1.0
Busca automática de metadados via YouTube API
Com API Key configurada, o sistema deve buscar e preencher automaticamente: título, canal, channelId, thumbnail, views, likes, commentCount, duração (ISO 8601), publishedAt, descrição, tags, youtubeCategoryId e liveBroadcastContent.
✍️RF09AltaVídeosv1.0
Preenchimento manual de vídeo
Sem API Key configurada, o sistema deve permitir ao usuário preencher manualmente ao menos o título do vídeo. Os demais campos são opcionais.
✏️RF10AltaVídeosv1.0
Editar vídeo existente
O sistema deve permitir editar as informações de um vídeo já salvo via updateVideo(id, updates). (Função disponível no hook useVideos, sem interface dedicada na v2.0 — não há botão “Editar” nos componentes de vídeo.)
🗑️RF11AltaVídeosv1.0
Excluir vídeo
O sistema deve permitir excluir permanentemente um vídeo individual via deleteVideo(id) ou múltiplos de uma vez via deleteVideos(ids[]).
↗️RF12AltaVídeosv1.0
Mover vídeo entre categorias
O sistema deve permitir mover um vídeo de uma categoria para outra, removendo-o da origem e adicionando-o ao destino via moveToCategory(videoId, fromCatId, toCatId).
📋RF13AltaVídeosv1.0
Copiar vídeo para outra categoria
O sistema deve permitir copiar um vídeo para outra categoria mantendo-o em ambas simultaneamente via copyToCategory(). Não duplica o objeto — apenas adiciona o ID da nova categoria ao array categories[].
🔀RF14AltaVídeosv1.0
Vídeo em múltiplas categorias
Um vídeo pode pertencer a múltiplas categorias simultaneamente. O campo categories[] armazena todos os IDs das categorias às quais pertence.
✅RF15AltaVídeosv1.0
Marcar vídeo como assistido
O sistema deve permitir alternar o campo watched (boolean) individualmente via toggleWatched(id) ou em lote via markMultipleWatched(ids[], watched).
⭐RF16AltaVídeosv1.0
Marcar vídeo como favorito
O sistema deve permitir alternar o campo favorite (boolean) via toggleFavorite(id). Favoritos ficam acessíveis na vista dedicada do dashboard.
☑️RF17AltaVídeosv1.0
Ações em massa sobre vídeos
O sistema deve permitir selecionar múltiplos vídeos e aplicar em lote: mover (moveMultipleToCategory), copiar (copyMultipleToCategory), excluir (deleteVideos) e marcar como assistido/não assistido (markMultipleWatched).
🔍RF18MédiaVídeosv1.0
Buscar vídeos por título ou canal
O sistema deve filtrar os vídeos de uma categoria por texto digitado, buscando apenas em título (case-insensitive) via filtragem inline em CategoryPage.
↕️RF19MédiaVídeosv1.0
Ordenar vídeos
O sistema deve permitir ordenar os vídeos por: data de adição, data de publicação, visualizações e duração — crescente e decrescente — via sortVideos() em utils/sortFunctions.js.
📡RF20MédiaVídeosv1.0
Verificar disponibilidade de vídeos
O sistema deve verificar em lote via checkVideosStatus() se os vídeos ainda estão disponíveis no YouTube, atualizando o campo videoStatus: 'available' (API) → null (modelo), 'unlisted' ou 'unavailable'.
👁️RF21BaixaVídeosv1.0
Preview flutuante por hover no card de vídeo
Na visão lista, ao passar o mouse sobre um VideoCard, o sistema deve exibir um painel flutuante (VideoPreview) com descrição, data de publicação e visualizações. O painel desaparece ao mover o mouse para fora do card.

🔄RF37MédiaDadosv2.0

🔀RF38MédiaDadosv2.0
Mesclar ou substituir ao importar
Na importação, o sistema deve oferecer: (a) mesclar — adiciona apenas itens com id ainda inexistente; ou (b) substituir — remove todos os dados atuais e carrega os importados.
🌐RF39BaixaDadosv2.0
Enriquecer vídeos importados via API

💾RF40BaixaDadosv2.0
Auto-backup via File System Access API
O sistema deve salvar automaticamente em arquivo local ao detectar mudanças nos dados, usando a File System Access API. Exibe status (conectado, salvando, último save, erro). Restrito a Chrome e Edge.

Zona de Perigo

🗑️RF41AltaDadosv2.1
Apagar todos os vídeos com recuperação
O sistema deve permitir apagar todos os vídeos de uma vez. A ação exige confirmação inline antes de executar. Após a exclusão, o sistema deve exibir um botão “Recuperar” com contagem regressiva de 10 minutos e barra de progresso visual. Se o botão for acionado dentro desse período, todos os vídeos são restaurados na íntegra. Após o prazo, os dados são perdidos permanentemente.
🗑️RF42AltaDadosv2.1
Apagar todas as categorias com recuperação
O sistema deve permitir apagar todas as categorias de uma vez, seguindo o mesmo fluxo de confirmação e recuperação de 10 minutos descrito em RF41. As duas operações (vídeos e categorias) são independentes e possuem janelas de recuperação próprias. Ambas ficam acessíveis na aba Perigo do SettingsModal.
🗑️RF43AltaDadosv2.1
Apagar apenas categorias vazias com recuperação
O sistema deve identificar automaticamente categorias que não possuem nenhum vídeo associado e permitir excluí-las em lote. A operação segue o mesmo fluxo de confirmação e janela de 10 minutos dos RF41/RF42. O botão fica desabilitado quando não existem categorias vazias. O snapshot de recuperação guarda a lista completa de categorias; ao recuperar, todas (inclusive as antes vazias) são restauradas.

Casos de Uso

Descrição detalhada das interações entre atores e o sistema.

Atores

👤 Usuário
Usuário final que acessa a aplicação via navegador. Interage com todas as funcionalidades.
🌐 YouTube API
Sistema externo. Fornece metadados de vídeos quando uma API Key válida está configurada.
UC01 Adicionar Vídeo por URL Alta
Ator principal Usuário
Pré-condição Pelo menos uma categoria existente no sistema.
Fluxo principal 1. Usuário clica em "Adicionar Vídeo".
2. Informa a URL do YouTube.
3. Sistema extrai o ID do vídeo da URL.
4. Se API Key configurada: sistema busca metadados automaticamente.
5. Usuário preenche / confirma os campos e seleciona a categoria.
6. Sistema salva o vídeo no LocalStorage.
Fluxo alternativo 4a. Sem API Key: usuário preenche os campos manualmente (título obrigatório).
Pós-condição Vídeo aparece na lista da categoria selecionada.
UC02 Criar Categoria Alta
Ator principal Usuário
Pré-condição Nenhuma.
Fluxo principal 1. Usuário clica em "Nova Categoria" no dashboard.
2. Informa o nome da categoria.
3. Sistema valida que o nome possui entre 2 e 50 caracteres e não é duplicado.
4. Sistema salva e exibe a nova categoria no grid.
Exceção Nome vazio, menor que 2 caracteres, maior que 50 caracteres ou já existente: sistema exibe mensagem de erro inline e não salva.
Pós-condição Nova categoria aparece no dashboard.
UC03 Exportar Dados Média
Ator principal Usuário
Pré-condição Pelo menos uma categoria ou vídeo cadastrado no sistema.
Fluxo principal 1. Usuário abre o SettingsModal e navega para a aba "Dados".
2. Clica em "Exportar JSON".
3. Sistema serializa categories e videos do localStorage via exportImport.js.
4. Browser dispara o download de um arquivo .json com timestamp no nome.
Pós-condição Arquivo .json salvo localmente; dados do app permanecem inalterados.
UC04 Importar Dados Média
Ator principal Usuário
Pré-condição Arquivo .json exportado previamente pelo sistema.
Fluxo principal 1. Usuário abre o SettingsModal e navega para a aba "Dados".
2. Usuário escolhe o modo de importação: "Mesclar dados" ou "Substituir tudo" (radio buttons exibidos antes do upload).
3. Clica em "Selecionar Arquivo" e escolhe o arquivo .json via file picker.
4. Sistema lê e valida a estrutura JSON (verifica campos obrigatórios de categories e videos).
5. handleImport() em App.jsx aplica migração de compatibilidade: vídeos com campo legado categoryId têm o campo renomeado automaticamente para youtubeCategoryId antes da persistência (compatibilidade com backups gerados antes da v2.0).
6. Sistema persiste os dados no localStorage e atualiza o AppContext.
Exceção Arquivo inválido ou mal-formado: sistema exibe Toast de erro e não altera os dados.
Pós-condição Dados importados visíveis no dashboard; histórico anterior preservado ou substituído conforme modo escolhido.
UC05 Mover Vídeos para Categoria Alta
Ator principal Usuário
Pré-condição Pelo menos um vídeo selecionado via checkbox; pelo menos duas categorias existentes.
Fluxo principal 1. Usuário seleciona um ou mais vídeos via VideoCheckbox na lista.
2. Clica em "Mover" no painel inline do CategoryActions (botão aparece quando há seleção).
3. Sistema abre MoveCopyModal com a lista de categorias disponíveis.
4. Usuário seleciona a categoria de destino e confirma.

Problemas e Soluções

Registro de desafios técnicos encontrados durante o desenvolvimento e como foram resolvidos.

Versão 1

🔴
P01 — Porta Vite dinâmica causa perda de dados do localStorage
✅
Solução: O localStorage é isolado por origem (protocolo + host + porta). Quando o Vite encontra a porta 3000 ocupada, automaticamente usa 3001, 3002 etc. — cada uma é uma origem diferente para o browser, tornando os dados previamente salvos invisíveis. O usuário perde acesso a todas as categorias e vídeos sem nenhum aviso. Resolvido adicionando strictPort: true ao vite.config.js: se a porta 3000 estiver ocupada, o Vite falha com erro em vez de trocar silenciosamente, forçando o desenvolvedor a liberar a porta antes de continuar.
🔴
P02 — React.memo com arrow function causa erro de compilação
✅
Solução: Ao envolver componentes com React.memo, a sintaxe React.memo(function Name({}) => {}) é inválida — mistura declaração de função nomeada com arrow function. A sintaxe correta é React.memo(function Name({ ...props }) { ... }) usando uma function expression nomeada. Alternativa válida: const Name = React.memo(({ ...props }) => { ... }). Este erro afetou VideoCard e VideoCardGrid durante a adição de otimizações de performance.
🔴
P03 — Ícone Loader inexistente na versão do lucide-react
✅
Solução: O componente Loader não existia inicialmente na versão lucide-react ^0.263.1 instalada, causando erro em tempo de execução. Após atualização, Loader passou a estar disponível e é atualmente usado em VideoForm, Button e Spinner. RefreshCw é usado em ImportData e MagicIdlePhase para ícones de atualização contextual. Recomendação: sempre verificar os ícones disponíveis na versão instalada via grep -r "export" node_modules/lucide-react/dist.
🔴
P04 — YouTube API retornava tags e categoryId mas não eram extraídos
✅
Solução: A resposta da YouTube Data API v3 (campo snippet) já inclui tags[] e categoryId, mas as funções fetchVideoData() em services/youtubeAPI.js e useYouTubeAPI.js não os incluíam no objeto retornado. Resolvido adicionando tags: video.snippet.tags || [] e youtubeCategoryId: video.snippet.categoryId || null ao retorno de ambas as funções (posteriormente consolidadas em P12). O useVideos.addVideo() também foi atualizado para persistir esses campos. Criado constants/youtubeCategories.js com mapeamento dos 17 IDs de categoria do YouTube para nomes em português. No VideoForm, após o preview, são exibidas sugestões clicáveis (pílulas) que permitem selecionar uma categoria existente ou criar automaticamente uma nova a partir da categoria/tags do YouTube.
🔴
P05 — Auto-Backup misturado na aba Dados causava confusão de UX
✅
Solução: O componente AutoBackup estava renderizado dentro da aba Dados do SettingsModal, acima de Exportar/Importar. Isso confundia o usuário, que não entendia a relação entre backup automático e export manual. Resolvido criando uma aba separada 🛡️ Backup no SettingsModal, movendo o AutoBackup para lá. A aba Dados voltou a conter apenas Exportar e Importar.
🔴
P06 — Encoding corrompido no vite.config.js com caracteres não-padrão
✅
Solução: O arquivo vite.config.js continha caracteres Â (non-breaking spaces, U+00A0) como indentação em vez de espaços normais. Isso impedia edições textuais automatizadas (replace_string_in_file) porque a string de busca não correspondia ao conteúdo real. Resolvido reescrevendo o arquivo inteiro com Set-Content -Encoding UTF8 e indentação limpa com espaços ASCII padrão.

Versão 2

🔴
P07 — File System Access API indisponível em Firefox e Safari
✅
Solução: O backup automático usa showSaveFilePicker() e FileSystemFileHandle, APIs exclusivas de browsers Chromium. Em Firefox e Safari a chamada lança TypeError: showSaveFilePicker is not a function. Resolvido com detecção em tempo de execução: const isSupported = 'showSaveFilePicker' in window. O componente AutoBackup renderiza uma mensagem explicativa quando isSupported === false, sugerindo Chrome/Edge e indicando que o export manual continua disponível para todos os browsers.
🔴
P08 — Dados do localStorage perdidos ao trocar porta ou browser no dev
✅
Solução: O localStorage é isolado por origem (protocolo + host + porta), portanto dados inseridos em localhost:3000 são invisíveis em localhost:5173 e desaparecem completamente ao limpar o cache do browser. Foi criado vite-plugin-local-db.js, um plugin Vite que expõe uma API REST local (GET/POST/DELETE /api/db) gravando em data/db.json no sistema de arquivos. O hook useLocalStorage passou a usar persistência dupla: grava em localStorage de forma síncrona e espelha no arquivo via fetch com debounce de 500 ms. Na montagem, o arquivo tem prioridade sobre o localStorage.
🔴
P09 — Importação de JSON no formato externo Tab URL Extractor
✅
Solução: Usuários vindos da extensão "Tab URL Extractor" exportam arquivos com estrutura { source: "tab-url-extractor", data: [ { url: "..." } ] }, incompatível com o formato nativo { categories: [], videos: [] }. Implementadas duas funções em services/exportImport.js: isTabExtractorFormat() detecta o formato pelo campo source e convertTabExtractorFormat() converte os dados — extraindo o videoId da URL, deduplicando entradas repetidas e descartando URLs que não são do YouTube — antes de passar o resultado pelo pipeline normal de importação.
🔴
P10 — Race condition entre carga assíncrona do arquivo e escrita imediata do usuário
✅
Solução: A carga inicial de data/db.json é assíncrona. Se o usuário adicionasse um vídeo antes da resposta chegar, o valor do arquivo sobrescrevia a mudança. Resolvido com a ref userHasWritten em useLocalStorage: na primeira chamada a setValue() a ref é marcada como true; ao chegar a resposta do arquivo, se userHasWritten.current === true o valor do arquivo é descartado, preservando o estado já modificado pelo usuário.
🔴
P11 — Testes assíncronos com erros em hooks React
✅
Solução: Ao testar hooks que lançam erros dentro de act(), o estado (setError, setIsLoading) apenas finaliza o flush se o erro for capturado dentro do callback de act(). O padrão correto é:

await act(async () => { try { await hook.method(); } catch(err) { thrown = err; } })

Capturar fora do act() impede que as atualizações de estado sejam processadas antes das asserções.
🔴
P12 — Duplicação de constantes e lógica entre service e hook
✅
Solução: O hook useYouTubeAPI reimplementava fetchVideoData, validateApiKeyFormat e testApiKey que já existiam em services/youtubeAPI.js. Refatorado para que o hook apenas gerencie estado React (isLoading, error) e delegue toda a lógica de rede para o service. O mesmo padrão foi aplicado a STORAGE_KEYS, parseDuration e formatDate.
🔴
P13 — React 18 Strict Mode executando side effects duas vezes dentro de setState
✅
Solução: Em React 18 Strict Mode o React pode invocar o updater de setState múltiplas vezes para detectar impurezas. Se a chamada a localStorage.setItem e ao fetch ficasse dentro do próprio updater, ocorriam gravações duplicadas (e avisos no console sobre estado pendente nos testes). Refatorado para que setValue seja um updater puro — apenas chama setStoredValue — e toda a persistência é movida para um useEffect que observa storedValue, garantindo que cada escrita ocorra exatamente uma vez após a renderização.
🔴
P14 — fetch não definido no ambiente de testes Node.js / jsdom
✅
Solução: useLocalStorage e useVideos chamam fetch() para sincronizar com data/db.json, mas fetch não existe nativamente em Node.js no contexto do jsdom utilizado pelo Vitest. Isso causava ReferenceError: fetch is not defined ao importar os hooks nos testes. Resolvido adicionando um mock global no topo de cada arquivo de teste que usa fetch (useLocalStorage.test.js, useVideos.test.js, useTheme.test.js, useCategories.test.js, youtubeAPI.test.js): global.fetch = vi.fn(() => Promise.resolve({ ok: true, json: () => Promise.resolve({ value: null }) })). Isso garante que a chamada ao arquivo retorna null de forma síncrona, sem alterar o comportamento testado do hook.
🔴
P15 — Campo categoryId legado impede categorização mágica pós-importação
✅
Contexto: A YouTube Data API v3 devolve o campo como snippet.categoryId. Em versões anteriores do app, esse valor era armazenado no vídeo sob o nome categoryId. O código foi posteriormente corrigido para usar youtubeCategoryId (evitando ambiguidade com o id de categoria interna), mas vídeos já salvos antes da mudança permaneceram com o campo antigo. Ao exportar e reimportar um backup, os 214 vídeos com categoryId ficavam sem youtubeCategoryId, tornando-os invisíveis para o algoritmo de categorização mágica (buildGroups()) e para os filtros baseados em youtubeCategoryId.

Solução: Adicionada migração automática em handleImport() no App.jsx: ao importar, cada vídeo que possui categoryId mas não youtubeCategoryId tem o campo renomeado on-the-fly antes de ser inserido no estado. O campo legado é removido e youtubeCategoryId é criado com o mesmo valor, garantindo compatibilidade total com backups antigos sem alterar o arquivo exportado.

Fix adicional: O método getBackupData() em App.jsx (usado pelo auto-backup via File System Access API) usava version: "1.0", enquanto exportAll() em services/exportImport.js usava "1.0.0". Padronizado para "1.0.0" em ambos, com a mesma ordem de campos: version, exportedAt, categories, videos.

P16 — Estado de recuperação da Zona de Perigo perdido ao fechar o modal
✅
Situação: O hook useDeletionWithRecovery era instanciado dentro do SettingsModal. Ao fechar o modal, o componente era desmontado e o estado do hook (snapshot + temporizador) era destruído. Ao reabrir o modal dentro da janela de 10 minutos, o botão “Recuperar” e a barra de progresso já não apareciam.

Solução: As três instâncias do hook (videosOp, categoriesOp, emptyCategoriesOp) foram movidas para App.jsx, que nunca é desmontado. Os ops são passados como props por App → SettingsModal → DangerZone. O estado de recuperação agora sobrevive a qualquer número de aberturas e fechamentos do modal enquanto a sessão estiver ativa.
